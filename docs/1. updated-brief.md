## Scenario

You are implementing a small User Management feature for an administration system.  
Administrators must be able to view users and enable or disable them.

> From a UI point of view, a simple table with a list of the 10 users sorted by 'name' (Surname) and an action button to enable/disable each user.
> There is an endpoint to return only active users. This is not mentioned in the UI spec, so add this as a feature after the critical 'all users' table is complete.

## Part 1 - Backend API (.NET)

Create a simple Web API with the following endpoints:  
<br/>GET /api/users  
Returns a list of users.  
<br/>GET /api/users/active  
Returns only active users sorted by Name.  
<br/>PUT /api/users/{id}/status  
Enable or disable a user.

Domain model:

public class User  
{  
public Guid Id { get; set; }  
public string Name { get; set; }  
public string Email { get; set; }  
public bool Active { get; set; }  
public List&lt;string&gt; Roles { get; set; }  
}

Seed the system with approximately 10 users.

> Lets keep it simple - we are showing architecture not specific implementation knowledge
> Storage: In-memory DB, One repository: UserRepository, One service: UserService, One Controller: UsersController
> Usually the repository, services and models would go into a Domain project, consider shortcutting this -- On reflection, it is not significant work to use a Domain project so this is not a sensible place to save time.


### Architecture Requirement

Structure the solution using separation of concerns, controllers should remain thin and not contain business logic.

### Business Rule

Users with the role "Admin" cannot be disabled.  
<br/>Return appropriate HTTP responses when rules are violated.

> At a minimum we will need 404 UserNotFound and ?422? AdminStatusNotEditable (not 403 as its not a permissions issue, but business logic.)
